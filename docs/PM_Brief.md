# PM_Brief：本地服装生图工作台（MVP 交接给 UI）

## 1. 一句话（面向业务）
把“上传衣服平铺图 +（可选）模特头像 + 需求参数”自动变成“可批量生成的 Shot List + 英文 Prompt”，并支持人工质检与“仅修衣服细节”的二次改图闭环；首版只在本机本地跑。

## 2. 背景与痛点（现状）
- 当前流程依赖人工：读图→策划→写结构化提示词→生图→反复修正
- 问题：耗时、质量不稳定、跑偏难定位、复用与沉淀弱

## 3. 目标（P0）
- 一键把需求转成结构化 Shot List（含每行可复制英文 Prompt）
- 逐张出图（按 Shot 一张一调），保证同任务统一性
- 人工质检：通过则结束；不通过则进入“对比→修复方案→改图”的闭环
- 修复原则：只改衣服细节；模特脸/姿势/场景/光线/镜头不改
- 全程可追溯：保留输入图、最终 Prompt、参数、输出图、修复版本

## 4. 主要用户与使用场景
- 角色：操作员（内部）
- 场景：小红书发图；风格（如：潮牌/高街）；统一大场景（如：街头）；N 张（如：4）

## 5. 范围边界
### In-scope（P0）
- 任务创建：上传衣服图（1~N）+ 可选 face_ref
- 需求表单：用途/平台、风格关键词、统一大场景、张数、动作偏好（可选）、约束（默认不新增 logo/不出现可读文字/不二维码/背景干净）
- 生成计划：输出结构化结果（含 GARMENT LOCK、Shot List、每行 Prompt）
- 计划可编辑：Shot 行、Prompt、以及 GARMENT LOCK（重点：可人工增删改）
- 逐张生图：每行 Shot 单独生成 1 张（更稳定、可重试）
- 质检与修复（人工触发）：生成修复方案（差异清单 + 修复指令草稿）→ 人工确认 → 执行改图
- 下载导出：单张/整任务打包导出
- 设置页：配置 API Key / Base URL / 默认输出比例与清晰度

### Out-of-scope（先不做）
- 多人协作/账号体系/权限隔离/云端部署
- 自动质检通过（P0 以人工为准；自动只做辅助）
- 复杂模板库/素材库/商品系统对接
- 成本/并发的精细化计费与预算中心（只做基础提示）

## 6. 成功标准（可验收）
- 操作员从“上传到拿到 4 张可用图”时长显著下降（以你们当前人工基线为参照）
- 生成结果可追溯可复跑（同一任务能定位到每次 Prompt/参数/图片版本）
- 修复闭环可用：能在不重生整张的情况下，把衣服关键细节拉回参考图

## 7. 关键产品决策（已确认）
- 本地单机 Web 工作台（无需登录）
- **Brain/Painter 分离**：策划用 Flash 2.0，生图用 Gemini 3 Pro (Image Preview)
- 4 张按“场景模板集合”批量产出（每张为一个 Shot）
- 人工质检：默认“通过”直接结束；“需要修复”才调用对比与改图
- GARMENT LOCK 支持全局共享；每个 Shot 允许追加 1~3 条覆盖项

## 8. 开放问题（不阻塞 UI 设计，但需在开发前补齐）
- Gemini 的实际网关域名/Base URL（你截图里平台的）
- API Key 的本地存储方式（文件/系统钥匙串/仅内存）
- “只改衣服细节”的成功率阈值与最大修复次数（建议 1~2 次）
